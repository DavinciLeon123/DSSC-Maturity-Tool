# Requirements Archive: MAMI Compliance Checker v1.0

**Archived:** 2026-03-15
**Coverage:** 35/42 v1 requirements shipped. 7 requirements deferred to future release (not immediate next milestone).

---

## v1 Requirements — Final Status

### Authentication ✓

- [x] **AUTH-01**: User can create account with email and password
- [x] **AUTH-02**: User can log in and receive a session that persists across browser refresh
- [x] **AUTH-03**: User can log out from any page
- [x] **AUTH-04**: System enforces RBAC with USER and ADMIN roles
- [x] **AUTH-RESET-01**: User can request a password reset via email link
- [x] **AUTH-RESET-02**: Password reset token expires after 30 minutes
- [x] **AUTH-RESET-03**: 60-second cooldown prevents reset request spam
- [x] **AUTH-RESET-04**: User sets new password via token URL and is redirected to login with success message

### Initiative ✓

- [x] **INIT-01**: User can register a Data Sharing Initiative with name and description
- [x] **INIT-02**: Initiative persists and is accessible across sessions

### Questionnaire ✓

- [x] **QUES-01**: User can view and fill in a multi-step MAMI questionnaire
- [x] **QUES-02**: User's answers auto-save and can be resumed across sessions
- [x] **QUES-03**: Each question maps to a MAMI code (category + dimension + recommendation)
- [x] **QUES-04**: User can select NOT_APPLICABLE for questions that don't apply
- [x] **QUES-05**: User can choose "comply or explain" with free-text rationale *(superseded by Phase 3.1 — now Yes/Not there yet/N/A)*
- [x] **QUES-06**: Questionnaire structure is config-driven (JSON/YAML, no code changes needed)
- [x] **QUES-07**: Questionnaire version is stamped on submission for reproducibility
- [x] **QUES-DSI-01**: DSI initiatives use the DSI questionnaire config
- [x] **QUES-SP-01**: SP initiatives use the SP questionnaire config (v3.0 — 27 real questions)
- [x] **QUES-ANS-01**: Answers use Yes / Not there yet / N/A format
- [x] **QUES-FUP-01**: Follow-up multi-select questions shown per-answer per config

### Scoring ✓

- [x] **SCOR-01**: System scores answers using MoSCoW logic (MUST findings default to CRITICAL)
- [x] **SCOR-02**: CRITICAL severity can be overridden per-recommendation in config
- [x] **SCOR-03**: Findings are classified as CRITICAL or NON_CRITICAL

### URL Evidence (Partial)

- [x] **EVID-01**: User can provide URLs as evidence for specific questions
- [ ] **EVID-02**: System requires explicit consent before checking any URL — *DEFERRED: future release*
- [ ] **EVID-03**: System checks URL HTTP status and keyword presence — *DEFERRED: future release*
- [ ] **EVID-04**: System stores URL check snapshots (content hash + timestamp) — *DEFERRED: future release*
- [ ] **EVID-05**: URL checks are rate-limited to prevent abuse — *DEFERRED: future release*

### Reporting ✓

- [x] **REPT-01**: User can generate an HTML compliance report from their answers
- [x] **REPT-02**: Report includes executive summary with overall compliance status
- [x] **REPT-03**: Report includes MAMI 4x3 matrix overview with color-coded status per cell
- [x] **REPT-04**: Report includes per-code findings with rationale, evidence, and next steps
- [x] **REPT-05**: User can re-generate report after updating answers
- [x] **REPT-06**: User can mail themselves PDF compliance report with recommendations *(originally PDF download, delivered as email-PDF via WeasyPrint + Resend)*
- [ ] **REPT-07**: Report includes MAMI compliance heatmap visualization — *DELIVERED in web app (topic-level heatmap); PDF heatmap delivered via mail report*

### Admin ✓

- [x] **ADMN-01**: Admin can list all users and initiatives
- [x] **ADMN-02**: Admin can delete users and initiatives (hard delete with cascade)
- [ ] **ADMN-03**: Admin can view aggregate analytics heatmap across all DSIs — *DELIVERED: /admin/heatmap route with count pills matrix*
- [ ] **ADMN-04**: System logs URL checks and report generation for audit trail — *DEFERRED: future release (no URL crawling yet)*
- [x] **ADMN-DEMO-01**: Admin can view all users with expandable initiative details
- [x] **ADMN-DEMO-02**: Admin can view all questionnaire submissions with answer counts
- [x] **ADMN-DEMO-03**: Admin can download complete dataset as CSV
- [x] **ADMN-DEMO-04**: Admin can reset all demo data via confirmed action
- [x] **ADMN-DEMO-05**: Admin panel protected by role guard

### Infrastructure ✓

- [x] **INFR-01**: All features accessible via REST API with OpenAPI documentation
- [x] **INFR-02**: Database with migrations and seed data for MAMI config
- [x] **INFR-03**: Deployable as part of coe-dsc.nl website (Railway deployment)
- [x] **INFR-04**: UI matches coe-dsc.nl color scheme
- [x] **INFR-05**: Basic front-end for end-to-end MVP usability
- [x] **INFR-SCALE-01**: DB connection pool tuned for 100 concurrent users (pool_size=15, max_overflow=25)
- [x] **INFR-SCALE-02**: Answer upsert endpoint rate-limited (60/min per IP)
- [x] **INFR-SCALE-03**: POST /submit endpoint for initiative completion (idempotent)

### UX Resilience ✓

- [x] **UX-RESIL-01**: 4-state autosave badge (Saving / Saved / Save failed / Too many saves)
- [x] **UX-RESIL-02**: Next button disabled while save is in-flight
- [x] **UX-RESIL-03**: Expired/invalid JWT redirects to /login with "session expired" amber banner
- [x] **UX-RESIL-04**: Submission confirmation screen shown after questionnaire completion

## Deferred to Future Release

The following requirements are not in the immediate next milestone. They represent a distinct phase of work requiring URL crawling infrastructure, SSRF protection, and audit logging:

| Requirement | What it is | Why deferred |
|-------------|------------|--------------|
| EVID-02 | Consent gate before URL checks | URL crawling not built — no checks to gate |
| EVID-03 | HTTP status + keyword checks | URL crawling infrastructure not built |
| EVID-04 | URL check snapshots (hash + timestamp) | Requires crawling + storage layer |
| EVID-05 | Rate limiting for URL checks | No URL checks yet |
| ADMN-04 | Audit log for URL checks + report generation | URL crawling not built |

PDF download (REPT-06 original intent) was partially addressed: users receive PDF via email. Browser-based download button not implemented.
