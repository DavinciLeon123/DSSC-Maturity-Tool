    Create config/dssc-questionnaire.json as a single file (D-10) containing all 6 categories and all 52 questions nested inside — no participant_type key, no DSI/SP split.

    Schema shape (plain data, no DSL):
    - Top-level keys: `version` (string, e.g. "dssc-v2-placeholder"), `default_options`, `categories`.
    - `default_options`: an array of exactly 5 objects, each `{ "label": <string>, "score": <int 1..5> }`, scores 1 through 5 in ascending order. This is the shared default 5-label set (D-09) reused by every placeholder question.
    - `categories`: an array of exactly 6 objects, each `{ "id": <string e.g. "cat-1">, "name": <placeholder string e.g. "Category 1">, "questions": [...] }`.
    - Distribute exactly 52 questions across the 6 categories summing to 52 (e.g. 9 + 9 + 9 + 9 + 8 + 8). The total across all categories MUST equal 52 (D-08).
    - Each question object: `{ "id": <globally-unique string e.g. "q-1-1">, "category_id": <matching parent category id>, "text": <placeholder string e.g. "Category 1 - Question 1"> }`. Placeholder questions omit an `options` key and inherit `default_options`.
    - Include per-question override support in the schema by documenting (in the config `version`/a top-level `_schema_note` string) that a question MAY carry its own `options` array of the same 5-object shape to override the defaults (QSTN-03). Demonstrate the override on exactly ONE question by giving it its own `options` array (5 objects, scores 1..5) so the loader/tests exercise both the default-inheritance and override paths.
    - Use generic placeholder text only (D-08); real content (QSTN-05) is a later content-authoring task requiring no schema change.
  </action>
  <verify>

  </verify>
  <done>File parses as JSON; exactly 6 categories; question count sums to exactly 52; default_options has 5 entries scored 1-5; at least one question carries an `options` override.</done>
