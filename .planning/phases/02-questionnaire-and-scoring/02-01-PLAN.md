          "category": "scheme",

At least one SHOULD-level code should exist per category to demonstrate non-critical defaults.

    MAMI code ID format: `{CATEGORY_PREFIX}-{DIMENSION_PREFIX}-{N}.{M}`
    - Category prefixes: S (scheme), PP (participants), D (data), SER (services)
    - Dimension prefixes: HRA (human_readable), MRA (machine_readable), TA (trust_anchors)
    - Example: S-HRA-1.1, S-MRA-1.1, PP-HRA-2.1, D-TA-3.1, SER-MRA-4.1

    **config/questionnaire-v1.json** — Questionnaire structure per research Pattern 2:
    ```json
    {
      "version": "1.0",
      "$schema": "https://json-schema.org/draft/2020-12",
      "categories": [
        {
          "id": "scheme",
          "label": "Scheme Management",
          "dimensions": [
            {
              "id": "human_readable",
              "label": "Human Readable/Actionable",
              "questions": [
                {
                  "id": "q_S-HRA-1.1",
                  "mami_code": "S-HRA-1.1",
                  "text": "Does your initiative have a publicly available governance document?",
                  "answer_type": "yes_no_explain",
                  "required": true
                }
              ]
            }
          ]
        }
      ]
    }
    ```
    Create questions for every code defined in mami-framework.json (one question per code minimum). Use `answer_type` values: `yes_no_explain` (supports COMPLY_EXPLAIN + rationale), `yes_no` (simple), or `not_applicable_allowed` (all options including NOT_APPLICABLE). Most MUST-level questions should use `yes_no_explain`. Add `not_applicable_allowed` to at least 3 questions to satisfy QUES-04.

    **config/scoring/mami-scoring.json** — JDM decision file per research Pattern 5. Use the SINGLE-ANSWER evaluation pattern (not array iteration — see research Pitfall 1). The JDM receives:
    ```json
    {"answer": {"mami_code": "S-HRA-1.1", "moscow_level": "MUST", "answer_value": "NO", "critical_override": null}}
    ```
    And returns:
    ```json
    {"severity": "CRITICAL", "status": "FINDING"}
    ```
    Structure: inputNode → decisionTableNode (hitPolicy: "first") → outputNode.
    Decision rules (in order, first-match wins):
    1. `critical_override == false` → severity: "NON_CRITICAL", status: "FINDING" (override wins regardless of MoSCoW)
    2. `critical_override == true` → severity: "CRITICAL", status: "FINDING"
    3. `moscow_level == "MUST"` AND `answer_value == "NO"` → severity: "CRITICAL", status: "FINDING"
    4. `moscow_level == "MUST"` AND `answer_value == "COMPLY_EXPLAIN"` → severity: "CRITICAL", status: "FINDING"
    5. `moscow_level == "SHOULD"` AND `answer_value == "NO"` → severity: "NON_CRITICAL", status: "FINDING"
    6. `moscow_level == "SHOULD"` AND `answer_value == "COMPLY_EXPLAIN"` → severity: "NON_CRITICAL", status: "FINDING"
    7. `answer_value == "YES"` → severity: "", status: "COMPLIANT"
    8. `answer_value == "NOT_APPLICABLE"` → severity: "", status: "NOT_APPLICABLE"

    Use ZEN Expression Language for conditions (strings require double-quoting in rules: `"\"MUST\""`, `"\"NO\""`). See research Pattern 5 for exact JSON structure with nodes, edges, and rules array format.

    Note: hitPolicy "first" is cleaner than "collect" for single-answer evaluation since rules are mutually exclusive.

    **Step 3: Create backend/app/services/mami_config.py**
    ```python
    """MAMI framework config loader utilities."""
    import json
    from pathlib import Path

    # Resolve config dir relative to this file's location:
    # services/ → app/ → backend/ → repo root → config/
    CONFIG_DIR = Path(__file__).parent.parent.parent.parent / "config"


    def load_mami_config() -> dict:
        """Load config/mami-framework.json. Returns full dict including version and codes list."""
        path = CONFIG_DIR / "mami-framework.json"
        return json.loads(path.read_text())


    def load_questionnaire_config() -> dict:
        """Load config/questionnaire-v1.json. Returns full dict including version and categories."""
        path = CONFIG_DIR / "questionnaire-v1.json"
        return json.loads(path.read_text())


    def get_scoring_dir() -> Path:
        """Return the path to config/scoring/ directory for ZEN engine loader."""
        return CONFIG_DIR / "scoring"
    ```
    Note on path resolution: `Path(__file__)` in `backend/app/services/mami_config.py` → `.parent` = services → `.parent` = app → `.parent` = backend → `.parent` = repo root → `/ "config"`. Verify this is correct relative to where uv runs the app (working dir = backend/). If working dir is `/app` in Docker, use `Path(__file__).parent.parent.parent.parent` which is repo root — this works regardless of working directory.

    **Step 5: Update backend/app/main.py** — Replace the bare `app = FastAPI(...)` instantiation with a lifespan-aware version. Keep ALL existing middleware and router registrations. The lifespan function loads configs and ZEN engine at startup:
    ```python
    from contextlib import asynccontextmanager
    import zen
    from app.services.mami_config import load_mami_config, load_questionnaire_config, get_scoring_dir

    @asynccontextmanager
    async def lifespan(app: FastAPI):
        # Load MAMI framework config
        app.state.mami_config = load_mami_config()
        # Load questionnaire config
        app.state.questionnaire_config = load_questionnaire_config()
        # Initialize ZEN Engine singleton with file-system loader
        scoring_dir = get_scoring_dir()
        def loader(key: str) -> str:
            return (scoring_dir / key).read_text()
        app.state.zen_engine = zen.ZenEngine({"loader": loader})
        yield
        # Shutdown: ZEN engine has no explicit close

    app = FastAPI(
        title="MAMI Checker API",
        ...,  # keep existing title/description/version/docs_url/redoc_url/contact
        lifespan=lifespan,
    )
    ```
    Important: Keep slowapi Limiter setup, CORSMiddleware, and all router includes EXACTLY as they are. Only change: add lifespan function + pass `lifespan=lifespan` to FastAPI constructor.

    Anti-pattern to avoid: Do NOT use `@app.on_event("startup")` — this is deprecated in FastAPI. The lifespan context manager is the correct approach.
