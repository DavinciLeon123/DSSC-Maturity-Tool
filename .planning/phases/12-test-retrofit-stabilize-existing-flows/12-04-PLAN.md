<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| report API → WeasyPrint | HTML → PDF rendering runs in-process; mocked in tests (never renders real PDFs / requires libpango in CI) |
| report API → Resend (email) | PDF report leaves the trust boundary via an external HTTP send; mocked in tests, never sent live |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12-04-SILENT | Denial of Service (silent failure) | `_send_report_email` bare `except Exception` (CONCERNS.md) | medium | mitigate | Tests assert the mocked `write_pdf`/`Emails.send` were actually called, so a background task silently swallowed by the bare-except fails the suite instead of passing green (Pitfall 3). The bare-except itself is fixed in Phase 18 (SECU-03) — characterized here, not fixed (D-04). |
| T-12-04-INFO | Information Disclosure | report content emailed to the authenticated user | low | accept | mail_report emails only to `current_user.email` (self); report content is the user's own assessment. No cross-tenant exposure. |
| T-12-04-AUTHZ | Elevation of Privilege | report endpoints check `initiative.user_id == current_user.id` | medium | mitigate | Ownership check is exercised implicitly by the happy-path tests (initiative owned by the authenticated user); a 404-on-foreign-initiative test may be added if trivial, otherwise deferred to Phase 18 hardening. |
| T-12-04-PDF | Tampering | WeasyPrint rendering untrusted HTML | low | accept | HTML is generated server-side from the user's own answers via Jinja2, not attacker-controlled markup; WeasyPrint mocked in tests. |
</threat_model>
